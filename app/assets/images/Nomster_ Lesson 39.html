Lesson 39: Adding an Image Upload Feature for Places

In this activity, you will add the ability to upload images to your Nomster application. We will guide you through adding a new database model and point you to the CarrierWave Ruby gem, which will make image uploading possible.

This is the most challenging self-guided lesson yet, but it's one of the most rewarding lessons to complete! It's challenging for a reason: you now have almost all the skills you need to tackle big web application features by yourself.

This lesson should cement many of the skills you've gained in the previous lessons and give you the ability to implement features without much guidance.

What would take this app to the next level would be a feature that allows a creator of a place to upload a few images of the place.

So, go ahead, be brave and awesome at the same time and add an image upload feature to your app. We have some hints for you below. Learning how to start tackling problems without everything spelled out will make you a much better coder.

Setting Up the Pipeline for Photos

Start by making a model that will capture your photos. Let's call the model photo:

$ rails generate model photo

Remember that generating a model will create a migration file, which lets Rails know what new table(s) to create in your database. Open your new photo migration file and add two fields to it: a text field called caption and an integer called place_id. If you can't remember the exact syntax, no worries, just refer back to how we did this with commentsLink opens in new tab. Run your migration from the terminal when you're ready.

This process also created an app/models/photo.rb file for our model. We'll need to update this model to establish its relationship to other models. Similar to comments, we want photos to belong to both users and places. That means the relationship will need to be defined in the User and Place models, as well.

Let's add a form on the page now to allow submitting new photos. Any time we're going to add a new form, though, we should also update the controller so the form has something to connect to. In app/controllers/places_controller.rb, take note of how we did this with comments and add another line to accommodate photos:

def show
  @place = Place.find(params[:id])
  @comment = Comment.new
end

Okay, now we're ready to actually add the form to the app/views/places/show.html.erb file. Note, however, that our Photo model only has a text field for the caption, so that's the only input we'll have. We'll worry about the actual image upload later. Use this code snippet of our previous comments logic but edit it to apply to photos:

<%= simple_form_for @comment, url: '#' do |f| %>
  <%= f.input :message %>
  <%= f.input :rating %>
  <%= f.submit "Add a Comment", class: 'btn btn-primary' %>
<% end %>

Remember how we used a # placeholder to prevent errors? This was because Simple Form couldn't find where to POST the data. We later solved this by creating a controller for comments. Similarly, we'll need to create a controller for photos. Use the terminal to do so, referring back to previous lessonsLink opens in new tab if you forget how (which is perfectly normal!).

Once created, this controller will need to be referenced in the config/routes.rb file. You can always run the rake routes command to verify if your configuration is correct. The URL should look like this in the end:

place_photos POST   /places/:place_id/photos(.:format) photos#create

This means we can go back and replace the # placeholder in our form's URL with something like place_photos_path(@place). That doesn't mean the form actually does anything, though. We still have to finish the photos_controller.rb file. Fortunately, the logic is going to be a lot like what we already did for comments. Use the code in comments_controller.rb to guide you, taking into account the names and properties we're using specifically for photos.

At this point, you'll know everything is working correctly if you can enter new photo captions on the Show Places page, and the browser redirects accordingly after each one. Of course, we still want to see these captions, so that's the next step. Use a Ruby loop (e.g. <% @place.photos.each do |photo| %>) to accomplish this.

Great job! Even though we're not displaying any images yet, you've laid some important groundwork to make it easier to implement this feature. Take a quick breather before moving on.

Making the Image Field Work

In this section you will be using a gem to help with the process of uploading an image into your application.

You have the option to write your own uploader function since you have already set up the pipeline in the previous section, having created the input fields, model, and controller. The next step involves the upload process which can get complicated due to many factors like using the subclass IO(Input Output Class), regarding file size dependencies, editing the content_type attribute containing the MIME type of the uploaded file, and also dealing with a multitude of potential tasks ranging from where to store the files (on disk, Amazon S3, etc) to creating thumbnails and resizing images.

Noting these intricacies are beyond the scope of this tutorial. Thankfully there are several libraries designed for this very purpose, so now that you know what an uploader library does for you under the hood, which one do you choose?

Using key factors to determine which library to pick is important because there are so many libraries that have similar functionality. It is a safe bet to select a library that has been used by many people which tells you it has withstood the test of time. Just think about a restaurant highly rated on Yelp. Also, picking a library that is currently maintained is important to provide some measure of confidence that this gem will not break in the near future.

For example look at the github repo for a popular library called CarrierWaveLink opens in new tab.

  • Functionality: README.md states that "this gems provides a simple and extremely flexible way to upload files from Ruby applications"
  • Popularity: Over 8,000 Stars & 90,000 Uses
  • Maintenance: More than 320 Contributors with fairly recent commits

This appears to be a solid option due to the high number of stars and recent commits. Install the gem and get started implementing this library into your app:

Step 1:
Let's loosely follow the Installation and Getting Started instructions located in the README.md of the CarrierWave repo hereLink opens in new tab and tweak the commands so they fit our application. First let's install the gem from the command line. 
$ gem install carrierwave
Step 2:
In Rails, add this to your Gemfile: 
gem 'carrierwave', '~> 1.0'
Finally, restart the server to apply the changes. As of version 1.0, CarrierWave requires Rails 4.0 or higher and Ruby 2.0 or higher. If you're on Rails 3, you should use v0.11.0.
Step 3:
Next you will create the uploader Picture by typing in the following command into the command line: 
$ rails generate uploader Picture
This should give you the following file: app/uploaders/picture_uploader.rb that should look something like this: 
class PictureUploader < CarrierWave::Uploader::Base
  storage :file
end
Step 4:
In the next two steps, you will connect the uploader to your model. First, you will add a string column to your model in app/models/photo.rb that will store the image or mount the uploader by creating a migration. 
rails g migration add_picture_to_photo picture:string 
rails db:migrate
Step 5:
Here you will modify the Photo model in order to mount the uploader: 
class Photo < ActiveRecord::Base
  mount_uploader :picture, PictureUploader
end
Step 6:
In the view with the "photo" form, follow the same pattern you used for the caption field.

HINT: Code that looks like this should go someplace: <%= f.input :picture %>.
Step 7:
Now that your photos form has a "picture" field, don't forget to adjust your controller in the photos_controller.rb file to capture this value.

HINT: You'll need to update the params method to include :picture somewhere.
Step 8:
Update the code that's currently listing all of the photo captions on the page to also display the picture. Note that if you simply use <%= photo.picture.url %>, it will print the URL directly (e.g. /uploads/1/2/848382.jpg) instead of the actual image. You'll need to use a Rails image_tag to convert it.

HINT: We've used the image_tag before when building the SplurtyLink opens in new tab app.

NOTE: If your app crashes at this point, it may be because we have incomplete data in the database from previous testing (i.e. photos with captions but no image URLs). If this happens, open the rails console and run Photo.delete_all to wipe out all photo data and start fresh.

Updating CarrierWave to Use S3

Once you get image uploading to work in your development environment, you'd actually run into some problems when you go to put this live on Heroku. According to HerokuLink opens in new tab, any files created after your app is built will get erased the next time the app is restarted. So uploading files directly to Heroku isn't a good solution. We'll need to use a third-party service to host our images. A popular choice for this is Amazon S3.

You should already have an AWS account, but if not, head over to the AWSLink opens in new tab homepage and sign up for free. Please note that you will need to enter credit card details, but Amazon will only charge your account if you exceed their monthly limits (which is highly unlikely).

Once your account is activated, navigate to the Amazon S3Link opens in new tab page and click on the blue Create Bucket button in the top-left corner:

In the dialog box that pops up, enter a unique bucket name (without periods), and select US East (North Virginia) for the region. Selecting this region is very important, because it is the default region that CarrierWave uses, and if you select a different region you will see errors in future steps.

After choosing a name and region, click Next. The next few screens will ask some aditional options, but leave the defaults for now and just keep clicking Next until the end, when you'll press the Create Bucket button to finish the setup process. If you had trouble following along, visit this screencastLink opens in new tab that performs the same actions.

The S3 bucket that you just created is where your image uploads will live once we're done. Next, though, we need to create a user in your AWS account who will have permission to use this bucket. Visit the IAM Management ConsoleLink opens in new tab and click the Add user button. You should see the following screen:

Give the user a name and set the Access Type to Programmatic Access. Then press the Next: Permissions button. The screen that follows is where we'll actually define the permissions for this user. Click on the third button that says Attach Existing Policies Directly. This will display a list of 400+ "policies" that this user could have, but the only one we care about is AmazonS3FullAccess. Search for this name and check the box next to it before continuing:

Keep clicking Next until you reach the end, then click Create user. A success page should display, listing the new user along with their Access key ID and Secret access key. Take note of these values. We will need them later. You can also optionally download these keys in a CSV file. Review this screencastLink opens in new tab if you had trouble getting this far.

Whew! With AWS configured, now we need to jump back into our Rails application. We want to switch CarrierWave over to using S3 while keeping our AWS keys safe. We did something similar back in the Google MapsLink opens in new tab lesson with our Geocoder API key. We can use the same figaro gemLink opens in new tab installation here. Open config/application.yml and add the following lines at the bottom, replacing the text with the values from your AWS account:

AWS_ACCESS_KEY: "enter-your-access-key-id"
AWS_SECRET_KEY: "enter-your-secret-access-key"
AWS_BUCKET: "enter-your-bucket-name"

Along with figaro, you'll need to install another gem called fog-aws that will ultimately use these keys to talk to AWS. After you've installed it, make a brand new file located at config/initalizers/carrierwave.rb with the following content inside:

# config/initializers/carrierwave.rb

CarrierWave.configure do |config|
  config.fog_provider = 'fog/aws'                        # required
  config.fog_public = false
  config.fog_credentials = {
    provider:              'AWS',                        # required
    aws_access_key_id:     ENV["AWS_ACCESS_KEY"],        # required
    aws_secret_access_key: ENV["AWS_SECRET_KEY"],        # required
  }
  config.fog_directory  = ENV["AWS_BUCKET"]              # required
end

Finally, we'll need to flip the switch with CarrierWave to use AWS instead of the local filesystem. Open app/uploaders/picture_uploader.rb and change the section about storage to look like this:

# storage :file
storage :fog

This will comment out, or disable, local file uploads in favor of AWS. You can always toggle this back to local files if you'd like. Restart your Rails server, though, and try out the app. If all's well, the photo feature should still work, but now images are being uploaded to your S3 bucket!

Of course, the whole point of this exercise was to make photo uploads work on Heroku, but if we push the code as-is, it will break, because Heroku doesn't know what your AWS keys are. Run the command bundle exec figaro heroku:set -e production to update Heroku, and you're all set!

  • Play with the CSS to make it look good.
